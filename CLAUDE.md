# ServerStack — Claude Code Instructions

## Project

Multi-provider server inventory & cost tracker. Self-hosted, Docker-deployed, AGPL-3.0.

## Tech Stack

- **Backend:** Node.js 22, Express 5, better-sqlite3, JWT auth, node-cron, nodemailer
- **Frontend:** React 19, Vite 6, Tailwind CSS 4, Recharts, Lucide React, react-i18next
- **Testing:** Vitest + Supertest (backend), Vitest + React Testing Library (frontend)
- **Deployment:** Single Docker container, SQLite via volume mount

## Code Rules

- ES Modules (import/export) throughout, no TypeScript — plain JS with JSDoc where needed
- Prepared statements for all SQL, never string concatenation
- Functional components with hooks, no classes
- Tailwind utility-first, no custom CSS except animations
- Every new function/fix MUST have unit tests. PRs without tests or with failing tests are NOT merged
- Commit messages: Conventional Commits (feat:, fix:, docs:, chore:)
- Commit messages NEVER contain "generated by Claude Code", "AI-generated", or similar attribution
- **Atomic commits: Each commit must represent ONE logical change.** Do NOT bundle multiple unrelated changes into a single commit. For example: creating a database schema is one commit, adding seed data is another, writing tests is another. A feature branch will have multiple commits — that's expected and desired. Squash merge handles cleanup when merging to development.

## Frontend Design

**Read /mnt/skills/public/frontend-design/SKILL.md before writing ANY frontend code.**

- MUST NOT look like generic AI design
- NEVER use Inter, Roboto, Arial for headings — pick distinctive fonts
- NEVER use purple-on-white gradients or cookie-cutter color schemes
- Dark mode is the DEFAULT theme
- CSS variables for all colors, consistent theming
- Micro-interactions and meaningful animations on hover states, transitions, loading
- Must feel like a premium developer tool, not a Bootstrap admin panel

## Internationalization (i18n)

- react-i18next for ALL user-facing text — no hardcoded strings in components
- `const { t } = useTranslation()` in every component with visible text
- Translation files by namespace: `i18n/{lang}/common.json`, `dashboard.json`, `servers.json`, `contracts.json`
- Languages: English (en) = default, German (de) = second
- `DEFAULT_LANG` env var controls initial language, user overrides persist in localStorage
- Date/number formatting respects selected locale
- Backend API errors returned as i18n keys, translated on frontend
- Every PR MUST include translation keys for both en and de

## Git Workflow

- **Never commit directly to main or development**
- Branch model: `main` (releases only) → `development` (integration) → `feat/*|fix/*|chore/*|docs/*` (feature branches)
- Feature branches branch from development, merge back via PR (squash merge)
- development → main only for releases via PR (regular merge)
- Every PR needs description: what changed, why, how to test
- One PR per feature/fix — no mega-PRs
- Delete feature branches after merge
- main must always be deployable

## CI/CD (GitHub Actions)

- `ci.yml` — Trigger: push/PR on development + feature branches. Runs lint, unit tests (backend + frontend), build check. MUST be green before merge.
- `docker-build.yml` — Trigger: push to main. Builds multi-arch Docker image (amd64 + arm64), pushes to ghcr.io. Tags: latest + version.
- `release.yml` — Trigger: tag push (v*). Creates GitHub Release using the merge PR description as release notes.

### Release Process

1. Create PR from development → main with changelog as PR description (from Conventional Commits, reviewed before merge)
2. Merge PR
3. Create tag: `git tag v1.x.x && git push --tags`
4. GitHub Actions builds container + creates release automatically

### Versioning (SemVer)

- MAJOR (v2.0.0) — breaking changes, API changes, DB migrations
- MINOR (v1.1.0) — new features, backwards compatible
- PATCH (v1.0.1) — bug fixes

## Data Model

6 tables: `providers`, `servers`, `contracts`, `ip_addresses`, `services`, `alerts`

### servers table — credential fields

| Column | Type | Notes |
|--------|------|-------|
| ssh_user | TEXT | SSH username |
| ssh_port | INTEGER | Default 22 |
| ssh_public_key | TEXT | Public key (reference only) |
| ssh_host_key | TEXT | Host key fingerprint for verification |
| login_user | TEXT | Optional login username |
| login_password_enc | TEXT | AES-256-GCM encrypted via ENCRYPTION_KEY env var |

**Private keys are NEVER stored.** Passwords encrypted at rest, decrypted only on demand (click-to-reveal), never in list API responses.

### contracts table — key fields

| Column | Type | Notes |
|--------|------|-------|
| monthly_cost | REAL | Current monthly cost |
| regular_cost | REAL | Regular price without discount |
| promo_price | BOOLEAN | Is this a promo? |
| promo_end_date | DATE | When promo ends |
| cancellation_period_days | INTEGER | Days notice required |
| next_cancellation_date | DATE | Next cancellation opportunity |
| auto_renew | BOOLEAN | Auto-renewal |

## API

REST, JSON, all under `/api/v1/`. JWT Bearer auth on all routes except auth endpoints.

Endpoints: providers (CRUD), servers (CRUD + services + ips), contracts (CRUD + /expiring), ip_addresses (CRUD), services (CRUD), dashboard (summary, costs, alerts, resources), export/import (JSON), auth (login, register, change-password).

## Testing

- Backend tests in `backend/__tests__/`
- Frontend tests in `frontend/__tests__/`
- In-memory SQLite for test DB, reset in beforeEach/afterEach
- Scripts: `npm test`, `npm run test:backend`, `npm run test:frontend`, `npm run test:watch`
- Test: API routes (happy + error), DB operations, auth flows, cost aggregation, alert logic, export/import, i18n rendering, crypto encrypt/decrypt

## README.md

Create a proper README.md with: project description, features list, screenshot placeholder, quick start (Docker), manual setup instructions, environment variables, tech stack, contributing guidelines, license badge (AGPL-3.0).

**The README must reflect the current state of the branch it lives on:**
- On `main`: README describes the latest stable release, links to Docker image on ghcr.io, shows stable installation instructions
- On `development`: README includes a note at the top that this is the development branch with unreleased features, lists what's new since last release
- On feature branches: no special README changes needed (inherits from development)

When creating a release PR (development → main), updating the README to reflect the new version is part of the PR.

## Prompt Sequence

1. **Project Setup + CI/CD** — Scaffold monorepo, create .gitignore, README.md (for development branch), .env.example, set up Vitest, i18n, Docker, GitHub Actions (ci.yml, docker-build.yml, release.yml), init git with main + development branches. **Set up GitHub repo rulesets via `gh` CLI:**
   - Ruleset "Protect main" on `main`: require PR before merging, require status checks to pass (ci), block force pushes, restrict deletions
   - Ruleset "Protect development" on `development`: require PR before merging, require status checks to pass (ci), block force pushes, restrict deletions
2. **Database** — All 6 tables, SSH + credential fields, crypto.js (AES-256-GCM), migrations, tests
3. **Auth** — JWT register/login/change-password, middleware, tests
4. **CRUD Routes** — All routes for providers/servers/contracts/ips/services, tests
5. **Dashboard API** — Summary/costs/alerts/resources endpoints, tests
6. **Frontend Layout** — Read frontend-design skill first. Sidebar, login, dashboard placeholders. Dark mode default. All strings via i18n.
7. **Server Pages** — ServerList, ServerForm (with SSH + credential fields), ServerDetail. i18n.
8. **Dashboard Widgets** — Cost display, server count, resources, donut chart, alert list. i18n.
9. **Costs & Contracts** — ContractList, CostAnalysis, trend chart, promo highlighting. i18n.
10. **Alerts & Export** — Alert cron job, in-app alerts, JSON export/import, settings (language switcher, DEFAULT_LANG). Tests.
